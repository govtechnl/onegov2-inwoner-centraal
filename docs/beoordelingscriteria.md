# Judging criteria

Teams are judged by an expert jury with domain expertise in digital
government services, AI, and technology. The criteria below come directly
from the challenge brief.

> **Tip.** Use these as a self-assessment checklist on Friday afternoon, not
> on Saturday morning. If you are missing a "Must" the day before the demo,
> change scope, not your sleep schedule.

## ✅ Must: minimum requirements for a valid submission

- [ ] **Useful to the bereaved partner.** The solution is demonstrably useful
      to the bereaved partner, not only to the government internally.
- [ ] **Works with the supplied data.** The prototype demonstrably works with
      the data in [`data/`](../data/) (or a self-composed variant in the same
      shape).
- [ ] **Multi-organisation linkage.** The overview links data from at least
      **two** government organisations.

## ⭐ Should: distinguishing qualities

- [ ] **Proactive.** The government initiates; the bereaved does not have to
      search.
- [ ] **Tone, timing, intelligibility.** The solution attends to *how* it
      communicates, not only *what* it shows.
- [ ] **Inclusive.** Accounts for low digital *doenvermogen* and limited
      language proficiency. (Use Anneke as a stress test,
      see [`personas.md`](personas.md#03--anneke-71--the-analogue-widow).)

## ⚠️ Should not: pitfalls to avoid

- [ ] **"Inside-out" solutions** that only help civil servants without any
      effect on the bereaved.
- [ ] **Greenfield rebuilds** that assume you can change every existing
      system or process. Reason from existing building blocks
      (see [`architectuur-context.md`](architectuur-context.md)).

## ✨ Could: bonus for outstanding submissions

- [ ] **Open source approach** with reusable components.
- [ ] **Alignment with existing tools** rijksoverheid.nl/overlijden,
      Berichtenbox, MijnOverheid.
- [ ] **Modular design** that scales to other life events: divorce,
      disability, retirement, becoming a parent, immigration.

## Deliverables

- Working prototype or demo (the core, even if not finished).
- 5-minute pitch + Q&A for the jury.
- Pitch deck, max. 10 slides.
- Repository link if code was written.
- Short description: what you built, for whom, why.

## Scoring rubric (suggested team self-score)

| Tier | Weight | Self-score (0–3) |
|---|---|---|
| Must - usefulness to bereaved | × 4 | |
| Must - works with data | × 3 | |
| Must - links ≥ 2 organisations | × 3 | |
| Should - proactive | × 2 | |
| Should - tone, timing, intelligibility | × 2 | |
| Should - inclusive | × 2 | |
| Could - open source / modular / aligned | × 1 | |

> Score 0 = absent, 3 = excellent. The jury will not use this exact rubric;
> it is a sanity check for your team.
